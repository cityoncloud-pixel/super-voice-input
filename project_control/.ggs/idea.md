# Idea

## 产品一句话与主轴（与 `newidea.md` 对齐）

超级语音输入器**不是**普通语音输入法，**也不是**单纯「录音转文字」工具，而是面向个人思考、写作、笔记、AI 对话、GAEH 工作流与编程任务的 **语音思想采集与文本投递工具**。

**核心价值**：用户可以用语音**分段**说出混乱、不完整、跳跃的想法；系统转写、整理、澄清，并按用户选择的 **本次场景**，将结果变成可用文本，**投递**到正确位置。

**设计主轴**（优于「只解决转写」）：

`本次场景 → 多段录音 → 分段转写 → 合并整理 → 输出投递`

多段 **`VoiceSession` + `VoiceSegment`** 是默认形态；单段录音不作为核心产品叙事。

# Goal: 构建支持多段录音会话的超级语音输入器

## Background

我经常在长文本输入时不想打键盘，尤其是在向 ChatGPT、Claude、Cursor、Codex、Claude Code、Obsidian 或 GAEH 输入较长想法时。普通输入法的语音输入只能把我说的话尽量原样转写出来，但我的真实需求不是简单转写，而是把一段或多段可能不连贯、重复、跳跃、语序混乱的口述整理成清楚、忠实原意、可直接使用的文本。

我之前做过一个录音转故事的程序，部署在服务器上，使用过豆包和 DeepSeek 两个 API 调用。但现在这个需求更日常、更底层，不应仅作为“录音转故事”功能存在，而应抽象成一个通用的语音输入增强工具。

更重要的是，我的真实思考过程往往不是一次录音就完整表达，而是会分多次补充、修正和追加。因此，本项目第一版不采用单段录音作为主流程，而是直接实现多段录音会话模式。

## Objective

构建一个本地优先的桌面端“超级语音输入器”，支持用户创建语音会话，在同一会话中录制多个语音片段。每个片段应独立保存、独立转写、可查看、可删除、可重录或重试。用户完成多段口述后，系统将所有片段的转写文本按顺序合并，并根据用户选择的整理模式生成最终可用文本。

核心流程为：

创建语音会话 → 录制多个语音片段 → 每段独立转写 → 合并转写文本 → 按模式整理 → 预览最终结果 → 复制输出 → 保存历史记录

## Core Positioning

本项目第一阶段不是完整智能体，不是语音操控电脑工具，不是会议纪要系统，也不是普通语音输入法。

它的第一身份是：

多段语音采集工具 + 语音转写工具 + 意图整理工具 + 文本输出工具

更准确地说，它是一个面向个人思考过程的语音输入基础设施。

未来它可以成为 Obsidian、GAEH、Cursor、ChatGPT、Claude、Codex 等系统的语音入口，但第一阶段必须专注完成“多段语音 → 可用文本”的稳定闭环。

## Main Requirements

1. 系统必须支持创建语音会话。
2. 一个语音会话必须支持包含多个录音片段。
3. 用户可以在当前会话中多次开始和结束录音。
4. 每次录音生成一个独立 VoiceSegment。
5. 每个 VoiceSegment 必须独立保存音频文件。
6. 每个 VoiceSegment 必须有顺序编号。
7. 每个 VoiceSegment 必须记录录音时长、创建时间、状态。
8. 每个 VoiceSegment 必须可以独立转写。
9. 某个片段转写失败，不应影响其他片段。
10. 用户可以查看每个片段的转写结果。
11. 用户可以删除某个片段。
12. 用户可以重试某个片段的转写。
13. 用户可以重录某个片段。
14. 用户点击“完成整理”后，系统应按顺序合并所有有效片段的转写文本。
15. 合并文本应保存为 combined_transcript。
16. 系统应根据用户选择的整理模式调用 LLM 生成 final_text。
17. 系统应保存 final_text。
18. 系统应显示最终结果预览。
19. 系统应支持复制 final_text 到剪贴板。
20. 系统应保存完整历史记录。
21. 系统应支持查看历史会话。
22. 系统应支持重新整理历史会话。
23. STT 调用必须通过 adapter 抽象。
24. Rewrite 调用必须通过 adapter 抽象。
25. Prompt 模板必须独立存放，不得硬编码在主流程中。
26. VoiceSession 和 VoiceSegment 必须是分离的数据结构。
27. 系统必须有统一错误处理。

## Recording Strategy

系统第一版应采用“分段转写，最终合并整理”的策略。

也就是说：

每段录音结束后，系统立即或手动触发该段转写；
每个片段保存自己的 raw_transcript；
用户完成全部录音后，系统合并所有片段的 raw_transcript；
最后调用整理模型生成 final_text。

第一版不建议采用“多个音频片段先合并为一个音频，再统一转写”的策略，因为该方式不利于失败恢复、片段删除、片段重试和局部管理。

## Required Modes（整理模式 · 基准）

本节定义 **系统层** 的六种 **`mode`**（prompt 通道）。**主界面** 不应长期并列暴露「场景预设 + 整理模式」两套一级选项；用户入口应收敛为 **「本次用途」**（用途预设 = `mode` + 默认投递），详见下文 **§界面与用途层级** 与 `goal.md` **G7**。

系统必须支持在 **高级设置或调试路径** 中直接选择 **整理模式**（以及会话级 `PATCH mode`）。产品规划明确：**不做** 自动判断模式、不做复杂 agent、不做自动 workflow routing。

### 核心目标（G6 已落地；G7 调整 UI 层级）

- **六种模式**各对应 **独立 prompt 模板**（`prompts/modes/*.md`）；整理按 **`session.mode`** 加载；prompt 正文不进前端。  
- **Mode Registry** + **`GET /modes`**；供高级设置或调试使用。  
- **VoiceSession** 持久化 **`mode`**；可在高级设置或 `PATCH` 后 **重新整理**（`combined_transcript` 不丢）。  
- **G7**：主路径用户选 **「本次用途」**，由 **用途预设** 自动写入 **`mode`** 与 **默认投递**；不再要求用户在首屏同时理解「预设」与「整理模式」。  
- **模式是业务语义**，与具体 LLM 厂商解耦；**整理引擎**仅按渲染后的 prompt 调用已配置的 Rewrite adapter。

### 六种模式（id · 名称 · 摘要）

| id | 名称 | 用途摘要 |
|----|------|----------|
| `clean_intent` | 原意清理 | 适合发给 ChatGPT、Claude、普通对话框；删口头禅与重复、修复断句与识别错误、忠实原意、不扩写观点 |
| `thinking_clarify` | **思考澄清**（新增） | 想法混乱、跳跃时：提炼真实意图，区分事实/困惑/判断/隐含问题，保留不确定性，给出下一步追问（**非简单润色**） |
| `obsidian_note` | Obsidian 笔记 | 输出可入库的结构化 Markdown（标题、摘要、要点、待办/后续问题） |
| `gaeh_goal` | GAEH Goal | 软件/流程类口述 → GAEH 可消费的 Goal（Background、Problem、Objective、Requirements、Non-goals、Acceptance Criteria 等） |
| `coding_task` | 编程任务 | 面向 Cursor/Codex/Claude Code 的**具体施工任务**（与 GAEH Goal 区分：Goal 偏目标定义，本模式偏可执行工程任务） |
| `faithful_transcript` | 忠实转录 | 尽量保留原始顺序与口吻；只轻度修正错字、标点、断句；不总结、不重组、不新增观点 |

**遗留别名**：API/SQLite 仍接受 `intent_cleanup`→`clean_intent`、`task_requirement`→`gaeh_goal`（启动迁移），而以表中 **canonical id** 为准。

### 设计约束摘要

- Prompt 模板落在服务端目录（推荐与仓库一致的 `prompts/modes/*.md`；与既有 `prompts/*.txt` 的并存与迁移在 PLAN 中约定）。  
- 模板渲染最低支持占位符 `{{combined_transcript}}`；可扩展 `session_title`、`mode_name` 等。  
- 后端统一加载：`mode → registry → 模板路径 → 读文件 → 渲染 → RewriteAdapter`；**adapter 内不写死各模式业务分支**。  
- 错误语义（对用户/日志）：`UNKNOWN_MODE`、`PROMPT_TEMPLATE_NOT_FOUND`、`PROMPT_RENDER_FAILED`。  
- 可选演进：新增 **`RewriteOutput`** 记录多次整理；短期可继续仅在 session 上保留最新 `final_text`，但实现上勿假设「一生只能整理一次」。

### 验收要点（功能 / 架构）

- **系统层**：六种 **mode** 均有独立模板；未知 mode / 缺模板时错误清晰；PromptLoader 与 RewriteAdapter 职责分离。  
- **用户层（G7）**：首屏 **用途** 选项与说明文案正确；选用途即绑定正确 **mode + 默认投递**；**思考澄清** 不作为泛「润色」推销。  
- 高级路径仍可切换 **mode** 并验证同一段 `combined_transcript` 得到不同结构输出。

## 界面与用途层级（Owner 定稿 · 纠正「双下拉」问题）

### 问题陈述（已实现 G6 后的 UX 负债）

当前工作台若 **并列** 暴露「场景预设」与「整理模式」两个一级下拉，会把 **内部架构概念**（preset / mode / 投递）同时摊在用户面前，导致：

- 用户分不清应先选预设还是选模式、二者是否冲突；  
- 「写入 Obsidian Inbox」与「Obsidian 笔记」等选项 **语义重叠**，增加认知负担；  
- 真实需求是 **「这次话要拿去干什么」**，而不是手动拼装 mode + 投递。

**结论**：不是撤销 mode 或预设机制，而是 **层级分离**——用户层只暴露 **用途**，系统层自动绑定 **整理模式 + 默认投递**。

### 两个层级（必须区分）

| 层级 | 用户是否需要理解 | 内容 |
|------|-------------------|------|
| **用户层** | 是 | **本次用途**（单一选择）：回答「我要把口述结果用在哪 / 哪种整理」。 |
| **系统层** | 默认否 | **`mode`（prompt 通道）**、**`output_target`（Output Router）**、引擎配置：由 **用途预设** 自动推导；仅在 **高级设置** 中允许覆盖。 |

**原则**：不要让用户在主界面组合「mode × preset」；**用途预设 = mode + default_output_target + 说明文案**。

### 主界面（**G7 已落地**；延伸见 **G8**）

1. **一级仅保留「本次用途」**（替代并列的「场景预设」「整理模式」）。  
2. 用途下方展示 **简短说明**（`description`），随选项变化。  
3. **标题** 可选；**录音区** 紧邻用途选择，降低路径长度。  
4. **转写引擎 / 整理引擎** 默认 **不占据首屏注意力**，放入 **「高级设置」** 折叠区（与可覆盖的 mode / 投递一致）。  
5. **删除或隐藏** 主路径上的「— 不使用预设 —」类开发者用语；若保留「裸配」能力，仅放在高级设置。

### 会话自动管理、双入口分工与状态机（**G8 目标**, `newidea.md`）

以下从战略文档 **`newidea.md`** 提炼为 **实现契约**（术语与代码对齐：**用途** = Registry 中 `id` / `use_case_id`；文中 **`preset_id`** 与之同义）。

#### 1）会话不必成为录音前的强制步骤

- **`VoiceSession` 仍是系统内部必要对象**，但体验上 **不要求** 用户每次先点「新建会话」再录音。  
- **正确行为**：用户点「开始录音」→ 调用 **`ensureCurrentSession`**（命名可映射到前端/IPC 逻辑）：若当前无可用会话，则 **按当前选中用途自动创建会话**（写入用途、`mode`、默认 `output_target`、合适 `status`），并 **立即开始录音**。  
- **「新建会话」** 保留为 **主动开始新主题** 的次要操作，不是高频主路径前置条件。  
- **无会话时的文案**：宜提示「点击开始录音将自动创建会话；也可先填标题再新建」，而非仅「尚未创建会话」。

#### 2）已完成会话再次录音 → 默认新开会话

- 若当前会话已为 **`completed`**（或等价「已终稿闭环」），用户再次点录音：默认应 **创建新会话**，避免新主题误追加到旧会话。  
- 若用户确需向旧会话追加：通过 **「追加补充」** 或工作台明确操作（后续迭代可加）。

#### 3）自动创建会话时的默认标题

建议规则：`{用途中文名} YYYY-MM-DD HH:mm`（与工作台可编辑标题并存）。

#### 4）切换用途（场景）时的规则

- **当前会话尚无片段**：可直接切换用途，并同步 **`use_case_id` / `mode` / 默认 `output_target`**。  
- **当前会话已有片段**：切换用途应 **提示**「将影响最终整理方式」；第一版可允许直接切换并同步上述字段；后续可增强为「仅本会话 / 新建会话并选用途」二选一。

#### 5）工作台 vs 悬浮窗分工（高频入口在悬浮窗）

| 层级 | 职责 |
|------|------|
| **工作台** | 完整控制台：用途与标题、会话与片段列表、每段转写、删除/重试/重录、完成整理、预览与投递、历史、API/路径/引擎配置、错误与调试。适合「认真整理与管理」。 |
| **悬浮窗** | 快速入口：**显示/切换当前用途**、开始录音、停止本段并转写、片段数量或状态摘要、完成整理、复制或投递、打开工作台。**不应**承载整理模式/引擎/复杂投递配置/完整历史。**不应**要求用户先打开主面板或先手动新建会话才能录音。 |

**产品判断**：日常高频路径应是 **悬浮窗**；大工作台是 **管理与调试台**，不是唯一主入口。

### 悬浮窗：遥控器，不是缩小工作台（**G9** · Owner 定稿）

当前悬浮窗已从「大工作台」推进到 **小型日常入口**，方向正确；下一步要从 **「功能压缩进小窗」** 升级为 **低干扰、状态驱动的高频悬浮输入器**。

#### 核心隐喻

- **悬浮窗 = 遥控器**：只承载 **录音 → 停段 → 整理 → 输出** 的高频动作与必要状态。  
- **主控台（大工作台）** = 完整片段编辑、历史、路径/API 配置、调试：不应要求用户每天在大界面完成主路径。

#### 须消除的体验负债（相对「缩小版工作台」）

- **信息密度**：片段区默认不宜铺满 **完整转写长文**；应以 **段号、状态、字数/时长、一行摘要** 为主，完整内容 **展开详情** 或跳转主控台。  
- **窗口 chrome**：Electron 悬浮窗默认 **`File / Edit / View / Window / Help`** 对小窗价值低、浪费高度；至少 **`autoHideMenuBar: true`**；并评估 **`frame` / `alwaysOnTop` / 固定或受限尺寸**，使形态接近「悬浮输入面板」而非普通文档窗口。  
- **文案用户化**：对用户不说「路由·剪贴板」类工程用语；宜 **「输出：剪贴板」「复制到剪贴板」「发送到：剪贴板」**。  
- **整理按钮与输出目标联动**：让用户知道「整理完之后去哪」。示例（与当前默认投递一致时）：剪贴板 → **整理并复制**；前台粘贴 → **整理并粘贴**；Obsidian/文件 → **整理并保存**；GAEH Goal 文件 → **整理为 Goal**；纯预览 → **生成终稿**（不自动投递或提示预览为主）。

#### 悬浮窗应长期保留（高频可见）

- 当前 **本次场景 / 用途**（可与记忆一致）、**当前输出目标**（用户可读）、**录音 / 停止本段**、**已录段数与各段状态（摘要）**、与输出目标联动的 **整理并发送**、必要时 **新建会话**（次要于录音与整理）、**打开主控台**。  
- **全局快捷键** 可保留，脚注 **压缩为一行**，勿长期占用大块区域。

#### 悬浮窗不应长期铺满（下放主控台或折叠）

- 片段 **全文**、长列表历史、STT/Rewrite **引擎选择**、复杂 **投递配置**：放在主控台或「展开详情」。  
- **完整终稿** 可在悬浮窗用短确认（如「已复制到剪贴板」）+ **查看详情** 打开主控台或折叠面板。

#### 状态驱动布局（客户端状态机 · 实现目标）

用户第一眼能区分当前处于：**等待输入 / 录音中 / 转写中 / 可整理 / 整理中 / 已完成 / 错误**。不同状态下 **主按钮与视觉焦点** 不同（录音时突出停段；可整理时突出「整理并×」）。  

建议客户端状态命名（与后端 `VoiceSession.status` 映射关系在 SPEC G9 约定，可渐进）：  
`idle`、`recording`、`transcribing`、`ready_to_finalize`、`finalizing`、`completed`、`error`。

#### G9 验收取向

- 默认 **不常驻系统菜单栏**（或等价隐藏）；用户能 **一眼读出** 当前状态与输出目标；片段区 **不因长文本撑爆窗口**；整理完成后 **明确反馈** 输出结果（与 G8 自动复制等行为一致时可合并文案）。

#### 6）主状态机（实现参考）

产品围绕状态驱动按钮启用逻辑，关键原则：**在 `idle_no_session` 下「开始录音」也必须可用**（依赖 §1 自动建会话）。

建议状态集合：`idle_no_session`、`draft_empty`、`recording`、`transcribing`、`draft_with_segments`、`rewriting`、`completed`、`error`（与现有字段可映射，不必一次性新增全部 DB 枚举）。**悬浮窗专属的状态细化与 UI 映射见上文 §G9。**

#### 7）投递与环境可用性

- **投递目标** 由场景默认 + 高级设置覆盖；**`foreground_paste`** 仅 Electron 可靠，不得作为唯一主路径；缺能力时应降级 **剪贴板** 并提示。  
- **路径类目标**（Markdown 目录、Obsidian Vault、GAEH 根）若未配置：宜 **`GET` 能力探测接口**（如 `GET /output-capabilities` 或 SPEC 约定路径）返回各 `output_target` **是否可用及原因**；UI **提前** 禁用或提示，避免用户操作后才遇 400。  
- **`final_text` 不因投递失败而丢失**（与 G1 一致）。

#### 8）Non-goals（重申）

不做复杂 agent；不做自动推断用户用途；不把整理模式与用途并列做主路径；不把前台粘贴作为唯一输出；不在 UI 硬编码 prompt 正文。

```text
继续说第二段
再次停止
按快捷键完成整理
系统生成文本
自动粘贴到当前输入框
```

用户感受到的不是“打开了一个软件”，而是：

> 我的电脑多了一个智能语音输入能力。

这个才是产品的核心方向。

## 流程 B：发送到 Obsidian

```text
按快捷键开始语音会话
说多段想法
选择/默认 Obsidian 笔记模式
完成整理
系统生成 Markdown
写入 Obsidian Inbox 或 Daily Notes
```

结果可以写到：

```text
ObsidianVault/00_Inbox/voice-2026-05-09-1430.md
```

或者：

```text
ObsidianVault/10_Daily/2026-05-09.md
```

这里建议先做 **写入 Inbox**，不要一开始改 Daily Notes，因为 Daily Notes 涉及追加位置、格式冲突、模板等问题。

---

## 流程 C：发送到 GAEH 项目

```text
用户在某个项目目录中
打开超级语音输入器的小入口
口述软件想法
选择“任务/需求整理”
系统生成 goal.md
保存到项目指定目录
```

例如：

```text
project/
  .gaeh/
    inbox/
      voice-goal-2026-05-09.md
```

或者：

```text
project/
  goals/
    voice-goal-super-input.md
```

这就把它变成了你 GAEH 的语音入口。

## 流程 D：发送到剪贴板

这是最低风险、最通用的第一步。

```text
语音 → 转写 → 整理 → 复制到剪贴板
```

然后你自己 Ctrl+V。

这个一定要先做稳，因为它兼容所有程序。

# 四、下一步不要继续美化大界面，应该先做“投递系统”

你现在最容易走偏的地方是继续优化这个大窗口 UI。
但我认为下一步重点不是美化，而是把输出结果变成“可投递对象”。

也就是说，系统内部要有一个明确的 Output Router。

# 五、建议新增核心模块：Output Router

当前流程大概是：

```text
final_text → 显示在页面里 → 用户复制
```

下一步应该改成：

```text
final_text → Output Router → 不同目标
```

建议抽象为：

```text
OutputTarget
  - clipboard
  - active_window
  - obsidian_file
  - gaeh_goal_file
  - markdown_file
  - custom_webhook
```

结构类似：

```python
class OutputTarget:
    def send(self, final_text: str, context: OutputContext) -> OutputResult:
        pass
```

# 六、输出目标优先级

我建议按这个顺序推进。

## P0：剪贴板输出

必须最先稳定。

```text
final_text → clipboard
```

验收标准：

```text
1. 生成终稿后自动复制或手动复制。
2. 复制后有明确提示。
3. 保留 final_text，不因复制失败丢失结果。
```

这是所有投递的底座。

---

## P1：当前输入框粘贴

这是最关键的高频能力。

```text
final_text → clipboard → 模拟 Ctrl+V → 当前焦点输入框
```

注意：这里不要一开始追求“识别 ChatGPT 输入框 DOM”这种复杂方式。

最简单、最稳的是：

```text
1. 保存当前焦点窗口
2. 生成 final_text
3. 写入剪贴板
4. 激活原窗口
5. 模拟 Ctrl+V
```

它对大多数程序都有效：

```text
ChatGPT 网页
Claude 网页
Obsidian
Cursor
VS Code
浏览器表单
微信/飞书输入框
普通文本框
```

这才是“超级语音输入器”的真正体验。

---

## P2：Obsidian 文件写入

```text
final_text → 指定 Vault/Inbox 目录 → markdown 文件
```

建议第一版配置：

```text
Obsidian Vault 路径
默认写入目录：00_Inbox
文件名规则：voice-YYYYMMDD-HHmm.md
```

不要一开始做复杂 Obsidian 插件。
先做普通文件写入就够了，因为 Obsidian 本质上就是 Markdown 文件夹。

---

## P3：GAEH 文件写入

```text
final_text → 当前项目 .gaeh/inbox/goal.md
```

这里可以支持一个“项目目录”配置：

```text
项目路径
输出类型：goal / task / bug_report / note
输出目录：.gaeh/inbox/
```

这个对你非常重要，因为它能把语音直接接入你的目标驱动开发流程。

---

## P4：Web/API 投递

后续再考虑：

```text
POST final_text 到本地 API
POST 到 GAEH runner
POST 到某个 agent service
```

这一步先不要急。

# 九、你现在应该让 GAEH 进入下一阶段：从 UI 面板到投递架构

可以给 GAEH 一个明确的新目标：

```markdown
# Goal: 将超级语音输入器从工作流面板升级为系统级文本投递工具

## 背景

当前系统已经实现或初步实现了多段语音会话、转写、整理和终稿预览。但现在主要使用方式仍是打开大界面，在界面中手动查看和复制结果。这不符合超级语音输入器的最终定位。

本工具的真正价值不是作为一个独立大窗口使用，而是作为系统级语音输入能力，将整理后的文本投递到用户正在使用的程序中，例如 ChatGPT、Obsidian、Cursor、浏览器输入框、GAEH 项目目录等。

## 目标

新增 Output Router 和多种输出目标，使 final_text 不仅能显示在界面中，还能被发送到指定位置。

## 主要需求

1. 新增 Output Router 模块。
2. 支持 ClipboardOutputTarget。
3. 支持 ActiveWindowPasteOutputTarget。
4. 支持 MarkdownFileOutputTarget。
5. 支持 ObsidianInboxOutputTarget。
6. 支持 GAEHGoalFileOutputTarget。
7. final_text 生成后可以选择输出目标。
8. 支持默认输出目标配置。
9. 支持生成后自动复制。
10. 支持生成后自动粘贴到原输入窗口。
11. 支持保存到 Obsidian Inbox。
12. 支持保存为 GAEH goal.md。
13. 输出失败不能导致 final_text 丢失。
14. 所有输出动作必须记录到 session history。
15. UI 中增加“输出到”选项，但不要让配置干扰主流程。

## 推荐输出目标优先级

P0：复制到剪贴板  
P1：粘贴到当前输入框  
P2：保存为 Markdown 文件  
P3：保存到 Obsidian Inbox  
P4：保存为 GAEH Goal 文件  
P5：POST 到本地 API 或 Agent 服务

## 验收标准

1. 用户生成终稿后，可以一键复制 final_text。
2. 用户可以选择“粘贴到当前输入框”。
3. 系统能将 final_text 写入剪贴板并模拟 Ctrl+V。
4. 用户可以配置 Obsidian Vault 路径。
5. 系统能将 Obsidian 模式输出保存为 Markdown 文件。
6. 用户可以配置 GAEH 项目目录。
7. 系统能将任务/需求模式输出保存为 goal.md 或带时间戳的 goal 文件。
8. 每次输出动作都有明确成功或失败提示。
9. 输出失败时 final_text 仍保留在会话中。
```

# 十一、最重要的架构原则

你要避免让 UI 组件直接决定输出逻辑。

不要这样：

```text
React Button → 直接写文件 / 直接粘贴 / 直接调系统 API
```

应该这样：

```text
React Button
  ↓
POST /api/sessions/{id}/outputs
  ↓
OutputRouter
  ↓
具体 OutputTarget
```

这样以后加 Obsidian、GAEH、Cursor、Webhook 都不会乱。
